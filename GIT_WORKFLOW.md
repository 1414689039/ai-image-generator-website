# GitHub 仓库操作规范

为了确保代码的安全性和版本控制的规范性，本项目严格执行以下 GitHub 操作规范。所有开发任务（包括功能开发、Bug 修复、重构等）完成后，**必须**遵循此流程进行提交。

## 1. 提交原则

*   **及时性**：每次完成一个独立的任务或修复后，立即提交代码，**严禁**在未提交代码的情况下结束当前对话或开始下一个任务。
*   **原子性**：每个 Commit 应该只包含一个独立的变更逻辑，避免将多个无关的修改混在一个 Commit 中。
*   **清晰性**：Commit Message 必须清晰地描述“做了什么”和“为什么做”。

## 2. 标准操作流程

在完成代码修改并验证通过后，按顺序执行以下命令：

### 第一步：检查状态
```bash
git status
```
确认当前有哪些文件发生了变更，确保没有遗漏或误添加的文件。

### 第二步：暂存更改
```bash
git add .
```
或者指定具体文件：`git add file_path`

### 第三步：提交更改 (Commit)
```bash
git commit -m "type: description"
```
**Type (类型) 必须遵循以下规范：**
*   `feat`: ✨ 新增功能 (feature)
*   `fix`: 🐛 修复 Bug
*   `docs`: 📚 文档修改
*   `style`: 💎 代码格式修改（不影响代码运行的变动）
*   `refactor`: ♻️ 代码重构（既不是新增功能，也不是修改bug的代码变动）
*   `perf`: 🚀 性能优化
*   `test`: 🚨 增加测试
*   `chore`: 🔧 构建过程或辅助工具的变动

**Description (描述) 要求：**
*   简明扼要地描述变更内容。
*   例如：`feat: Add gallery waterfall layout` 或 `fix: Resolve message board crash on empty list`

### 第四步：推送到远程 (Push)
```bash
git push origin main
```
*注：如果当前不在 main 分支，请先切换回 main 分支并合并，或者推送到对应的远程分支。*

## 3. 安全注意事项

*   **敏感信息检查**：在 `git add` 之前，务必检查代码中是否包含 API Key、Secret、Password 等敏感信息。敏感信息应存放在 `.env` 文件或环境变量中，并确保 `.env` 已被加入 `.gitignore`。
*   **冲突解决**：如果在 `push` 时遇到冲突，必须先 `pull` 远程代码，手动解决冲突并验证无误后，再次提交推送。
